# 개발 규칙

## 승인 규칙

코드, 에셋, Config, 빌드 파일, 프로젝트 설정은 사용자에게 명시적으로 승인받기 전에는 수정하지 않습니다.

이런 파일을 수정해야 할 때는 먼저 아래 내용을 설명합니다.

- 어떤 파일을 바꿀 것인가.
- 왜 그 변경이 필요한가.
- 가장 작고 안전한 구현은 무엇인가.
- 어떻게 검증할 것인가.

## 작업 순서

새 기능을 시작할 때는 아래 순서를 사용합니다.

1. 관련 문서를 읽거나 갱신합니다.
2. 기능의 소유 폴더/모듈을 확인합니다.
3. 가장 작은 변경 단위를 제안합니다.
4. 승인을 받습니다.
5. 구현합니다.
6. 빌드하거나 테스트합니다.
7. 개발 로그를 갱신합니다.

## 프로젝트 구조 방향

숨겨진 동작보다 명시적인 시스템을 선호합니다.

적합한 경우 Unreal의 데이터 기반 워크플로우를 우선합니다.

- 상태와 카테고리 식별에는 Gameplay Tags를 사용합니다.
- 조정 가능한 게임플레이 정의에는 Data Assets를 사용합니다.
- 입력 매핑에는 Enhanced Input을 사용합니다.
- 능력, 효과, 쿨다운, 속성이 충분히 복잡해지면 Gameplay Ability System을 사용합니다.
- UI가 prototype 범위를 넘어서면 UMG/CommonUI 스타일의 레이어 구조를 사용합니다.

## 프로토타입 규율

빠르게 prototype을 만들 수는 있지만, 구조가 불분명하게 남아서는 안 됩니다.

임시 prototype에도 아래 항목은 필요합니다.

- 명확한 폴더 위치.
- 명확한 에셋 이름.
- 명확한 소유 영역.
- `Docs/04_Logs/Development_Log.md`의 기록.

## 네이밍 규율

이름은 에셋 타입, 도메인, 목적을 전달해야 합니다.

피해야 할 이름:

- `NewBlueprint`
- `TestActor`
- `TempMap`
- `MyWidget`

임시 파일은 `TMP_` 접두사를 쓰거나, 명확한 `_Prototype` 폴더 안에 둡니다.

## 숨겨진 아키텍처 금지

중요한 규칙은 `Docs/`에 기록합니다.

결정이 바뀌면 관련 문서를 수정하고 개발 로그에 남깁니다.
