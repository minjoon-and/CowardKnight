# Git 컨벤션

## 목적

이 문서는 CowardKnight 프로젝트의 Git 커밋 메시지 작성 규칙을 정의합니다.

목표는 다음과 같습니다.

- 커밋 기록만 보고도 변경 의도를 빠르게 파악할 수 있게 합니다.
- Unreal 프로젝트 특성상 코드, 에셋, Config, 문서 변경을 구분합니다.
- 1인 개발 프로젝트라도 추후 협업과 LLM 작업 추적이 가능하도록 기록을 남깁니다.

## 기본 형식

커밋 메시지는 아래 형식을 사용합니다.

```text
type: 짧은 제목

변경사항
* 핵심 변경 1
* 핵심 변경 2
* 핵심 변경 3
```

본문의 `변경사항` 항목은 최대한 5개를 넘기지 않습니다.

5개를 넘겨야 한다면 커밋이 너무 큰지 먼저 확인합니다.

## 제목 규칙

제목은 한 줄로 작성합니다.

```text
type: 무엇을 왜 바꿨는지 짧게 요약
```

규칙:

- 제목은 50자 안팎을 목표로 합니다.
- 마침표를 붙이지 않습니다.
- 너무 포괄적인 제목을 피합니다.
- 가능하면 한글로 작성합니다.
- Unreal 고유명사, 클래스명, 에셋명은 원문을 유지합니다.

좋은 예:

```text
docs: 프로젝트 문서와 폴더 규칙 정리
feat: Bolt 기본 이동 입력 추가
fix: 레벨업 카드 선택 중 입력 잠금 누락 수정
content: Stage01 화이트박스 맵 추가
config: 기본 GameMode 설정 변경
```

피해야 할 예:

```text
update
fix bug
작업함
여러 가지 수정
```

## 커밋 타입

이 프로젝트는 Conventional Commits 스타일을 기반으로 하되, Unreal 프로젝트에 맞게 일부 타입을 추가합니다.

| 타입 | 사용 시점 |
| --- | --- |
| `feat` | 새로운 플레이 기능, 시스템, 유저가 체감하는 기능 추가 |
| `fix` | 버그, 오동작, 크래시, 잘못된 게임플레이 수정 |
| `docs` | Markdown 문서, README, 개발 로그, 기획 문서 수정 |
| `style` | 동작 변경 없는 코드 스타일, 포맷, 네이밍 정리 |
| `refactor` | 동작 변경 없이 구조 개선 |
| `test` | 테스트 코드, 검증용 자동화, 테스트 데이터 추가/수정 |
| `build` | 빌드 시스템, 모듈 의존성, Target, Build.cs 변경 |
| `config` | Unreal Config, 프로젝트 설정, 에디터 설정 변경 |
| `content` | Unreal 에셋, 맵, Blueprint, Material, VFX, Animation 변경 |
| `asset` | 외부 에셋 import, 에셋 정리, 원본 리소스 추가 |
| `chore` | 개발 보조 작업, 폴더 정리, 메타 작업 |
| `revert` | 이전 커밋 되돌리기 |

## 타입 선택 기준

여러 타입이 섞이면 커밋을 나누는 것을 우선합니다.

나누기 어렵다면 변경의 주된 목적을 기준으로 타입을 선택합니다.

예:

- 문서만 바꿨다 → `docs`
- C++ 코드 기능을 추가했다 → `feat`
- Blueprint 에셋만 수정했다 → `content`
- `.uproject`, `.ini`, `Build.cs`를 바꿨다 → `config` 또는 `build`
- 폴더 구조와 `.gitkeep`만 추가했다 → `chore`

## 본문 규칙

본문은 선택 사항이지만, 변경이 2개 이상이면 작성합니다.

본문은 아래 형식을 사용합니다.

```text
변경사항
* 첫 번째 핵심 변경
* 두 번째 핵심 변경
* 세 번째 핵심 변경
```

규칙:

- 항목은 최대 5개를 목표로 합니다.
- 각 항목은 한 줄로 짧게 씁니다.
- 구현 세부보다 결과와 의도를 적습니다.
- 너무 작은 변경을 모두 나열하지 않습니다.
- "수정함", "추가함"만 반복하지 말고 대상과 이유를 함께 씁니다.

좋은 예:

```text
feat: Bolt 기본 이동 입력 추가

변경사항
* Bolt 캐릭터에 좌우 이동과 점프 입력을 연결
* SideViewCamera와 CameraBoom 기본값 설정
* 기본 GameMode에서 Bolt Pawn을 사용하도록 지정
```

좋은 예:

```text
docs: Git 커밋 컨벤션 추가

변경사항
* Conventional Commits 기반 타입 규칙 정리
* Unreal 전용 content, asset, config 타입 추가
* 커밋 본문을 최대 5개 핵심 항목으로 제한
```

## Unreal 프로젝트 커밋 주의사항

Unreal 프로젝트는 바이너리 에셋이 많기 때문에 커밋 단위를 작게 유지합니다.

권장:

- C++ 코드와 `.uasset` 변경을 가능하면 분리합니다.
- Config 변경은 별도 커밋으로 분리합니다.
- 대형 에셋 import는 `asset` 커밋으로 따로 남깁니다.
- 맵 변경은 `content` 커밋으로 따로 남깁니다.
- 문서 변경은 `docs` 커밋으로 분리합니다.

피해야 할 것:

- 코드, 맵, 에셋 import, Config, 문서를 한 커밋에 모두 섞기.
- Unreal Editor가 자동 변경한 파일을 확인 없이 함께 커밋하기.
- 의미 없는 중간 저장 커밋을 오래 남기기.

## 커밋 전 체크리스트

커밋 전 아래를 확인합니다.

- `git status`로 변경 파일을 확인했습니다.
- 의도하지 않은 Config 또는 에셋 변경이 없는지 확인했습니다.
- 커밋 타입이 변경 목적과 맞습니다.
- 제목이 너무 넓거나 모호하지 않습니다.
- 본문의 변경사항은 5개 이하의 핵심 항목입니다.
- 커밋 메시지에 AI 공동 작성자·생성 표기가 없습니다.

## 작성자 표기

커밋은 개발자 단독 작성으로 기록합니다.

- 커밋 메시지에 `Co-Authored-By` 트레일러를 넣지 않습니다.
- "Generated with", AI 도구 이름 등 생성 출처 표기를 넣지 않습니다.
- LLM 에이전트가 커밋을 만들 때도 이 규칙을 따릅니다.

## LLM 에이전트 규칙

사용자가 아래와 같이 요청하면 LLM은 반드시 이 문서를 기준으로 답변합니다.

- "깃 메시지 추천해줘"
- "커밋 메시지 추천해줘"
- "commit message 추천"
- "git commit 메시지 뭐가 좋아?"

LLM이 커밋 메시지를 추천할 때는 아래 형식을 우선합니다.

```text
type: 제목

변경사항
* 핵심 변경 1
* 핵심 변경 2
* 핵심 변경 3
```

LLM은 변경사항을 길게 설명하지 않습니다.

최대 5개 항목으로 요약합니다.

작업 범위가 섞여 있으면 커밋을 나누는 방안을 먼저 제안합니다.

## 추천 답변 방식

사용자가 단순히 커밋 메시지만 요청했다면 긴 설명 없이 바로 추천 메시지를 제시합니다.

권장 답변:

```text
추천 커밋 메시지:

docs: Git 커밋 컨벤션 추가

변경사항
* Conventional Commits 기반 타입 규칙 정리
* Unreal 전용 content, asset, config 타입 추가
* 커밋 본문을 최대 5개 핵심 항목으로 제한
```

변경 범위가 넓어서 커밋을 나누는 편이 좋다면 아래처럼 제안합니다.

```text
이번 변경은 커밋을 나누는 편이 좋습니다.

1. docs: 프로젝트 문서 체계 정리
2. chore: Source와 Content 기본 폴더 구조 추가
```
